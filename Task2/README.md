# Задание 2. Проектирование RTB и интеграции с DSP

## Что нужно сделать

Спроектировать архитектуру критичного компонента — **RTB-сервиса** для интеграции с новой DSP-площадкой с соблюдением требований по времени отклика и надёжности.

---

## Конкретные задачи

### 1. Проектирование сервиса ставок

Необходимо:

- выделить сервис ставок из монолита;
- описать его границы и API;
- определить зависимости от других сервисов;
- спроектировать модель данных сервиса.

---

### 2. Проектирование взаимодействия

Необходимо определить протоколы и паттерны взаимодействия:

- для внутреннего взаимодействия:
  - REST;
  - gRPC;
  - асинхронный подход;
- для внешнего взаимодействия:
  - протокол интеграции с DSP;
- для потоковой обработки событий:
  - Kafka для событий кликов и показов.

Также необходимо обосновать:

- почему использовать gRPC или REST для взаимодействия `Bidding` и `Campaign`;
- почему Kafka подходит для событий;
- нужен ли API Gateway;
- если API Gateway нужен — какой именно.

---

### 3. API Gateway для DSP-интеграции

Необходимо спроектировать API-слой для новой DSP-площадки.

API Gateway должен покрывать:

- маршрутизацию запросов на ставки;
- ограничение частоты запросов;
- аутентификацию;
- паттерн автоматического размыкателя, то есть Circuit Breaker;
- мониторинг времени отклика.

---

### 4. Паттерны надёжности

Необходимо описать, как обеспечить надёжность системы.

Следует рассмотреть:

- паттерн **Circuit Breaker** для вызовов зависимых сервисов;
- повторные попытки с экспоненциальной задержкой;
- идемпотентность финансовых операций;
- резервные стратегии:
  - кэшированные ставки;
  - дефолтные ставки.

---
