# Выбор архитектурного стиля для эволюции AdScale

## 1. Доводы в пользу отказа от монолита

Оставить текущий монолит без архитектурных изменений нельзя, потому что он уже не удовлетворяет ключевым бизнес- и техническим требованиям.

### 1.1. Монолит уже нарушает SLA

| Метрика | Текущее значение | Требование |
|---|---:|---:|
| Пиковая нагрузка | 15 000 RPS | 15 000 RPS сейчас |
| P95 latency | 110–120 ms | ≤ 100 ms |
| Новая нагрузка после DSP | 18 000 RPS | +20% |
| Целевой latency DSP | — | ≤ 80 ms, жёсткий SLA партнёра |

---

### 1.2. Монолит не позволяет масштабировать компоненты независимо

В AdScale разные компоненты имеют разный профиль нагрузки:

| Компонент | Профиль нагрузки |
|---|---|
| Ad Server | Высокий RPS, latency-sensitive |
| Auction Engine | Critical path, CPU/latency-sensitive |
| Statistics Service | Высокая write-нагрузка |
| Analytics Service | Тяжёлые read-запросы |
| Financial Service | Транзакционная консистентность |
| Dashboard | Пользовательские CRUD-запросы |

---

### 1.3. Монолит смешивает критичные и некритичные потоки

Сейчас в одной архитектурной области конкурируют за общую PostgreSQL:

| Поток / область | Критичность |
|---|---|
| RTB-аукцион | Критичный latency-sensitive поток |
| Запись показов | Высокая write-нагрузка |
| Запись кликов | Высокая write-нагрузка |
| Аналитика | Тяжёлые read-запросы |
| Отчётность | Некритичный пользовательский поток |
| Финансы | Транзакционная консистентность |
| Операции dashboard | CRUD-нагрузка |

**Следствие:** некритичный отчёт рекламодателя может ухудшить работу критичного RTB-аукциона.

---

### 1.4. Монолит усиливает риск отказа

Текущая архитектура имеет несколько single points of failure:

| Single point of failure | Проблема |
|---|---|
| Одна PostgreSQL | Общий bottleneck и единая точка отказа |
| Один production-кластер | Нет изоляции окружений и отказоустойчивости |
| Ручной деплой | Высокий риск ошибок при релизах |
| Отсутствие HA | Нет высокой доступности |
| Отсутствие контейнеризации | Сложнее масштабировать и воспроизводить окружения |
| Отсутствие очередей | Нет буферизации и асинхронной обработки |

**Вывод:** монолитная структура увеличивает радиус поражения: ошибка в одном компоненте может повлиять на всю платформу.

---

### 1.5. Монолит не соответствует целям на год

| Цель | Почему текущий монолит не подходит |
|---|---|
| 50 000 RPS | Нет горизонтального масштабирования hot path |
| Event-driven обработка | Нет очередей и событийной модели |
| Микросервисная архитектура | Текущий монолит не декомпозирован |
| Мультирегиональность | Один production-кластер и одна БД |
| SLA 99,9% | Нет HA и отказоустойчивой архитектуры |
| Мульти-DSP интеграции | Нет изоляции партнёрских потоков |

# 2. Сравнение вариантов архитектурного стиля

## Вариант 1 — микросервисы

| Критерий | Оценка |
|---|---|
| Скорость изменений | Низкая |
| Риск внедрения | Очень высокий |
| Масштабируемость | Потенциально высокая |
| Latency | Может ухудшиться из-за сетевых вызовов |
| Отказоустойчивость | Требует зрелой платформы |
| Подходит для 3 месяцев | Нет |
| Подходит для 1 года | Да, как целевое направление |

**Вывод:** микросервисы подходят как целевая архитектура, но не как немедленный rewrite.

---

## Вариант 2 — модульный монолит

Модульный монолит предполагает, что система остаётся единым deployment unit, но внутри неё вводятся чёткие границы модулей.

| Плюсы | Минусы |
|---|---|
| Меньше операционной сложности | Не решает полностью проблему независимого масштабирования |
| Быстрее внедрить, чем микросервисы | Не устраняет общий bottleneck PostgreSQL |
| Помогает навести порядок в коде | Не даёт полноценной fault isolation |
| Снижает связанность | Не позволяет масштабировать Auction Engine отдельно, если deployment остаётся общим |
| Готовит систему к будущей декомпозиции | — |

**Вывод:** модульный монолит полезен как промежуточный этап, но недостаточен сам по себе для целей AdScale.

---

## Вариант 3 — Strangler Fig Pattern и постепенная декомпозиция

**Strangler Fig Pattern** — это постепенная замена частей старой системы новыми сервисами через перехват или маршрутизацию отдельных потоков.

### Идея подхода

| Шаг | Суть |
|---|---|
| 1 | Старый монолит продолжает работать |
| 2 | Новые функции или критичные потоки выносятся рядом |
| 3 | Трафик постепенно переключается на новые сервисы |
| 4 | Старые части монолита постепенно отключаются |

### Плюсы и минусы

| Плюсы | Минусы |
|---|---|
| Минимизирует риск big-bang rewrite | Временно возникает гибридная архитектура |
| Позволяет делать миграцию поэтапно | Нужны адаптеры и маршрутизация |
| Даёт быстрый эффект на critical path | Требуется дисциплина в управлении границами |
| Позволяет постепенно выделять сервисы | Есть риск дублирования логики на переходном этапе |
| Сохраняет работоспособность монолита | — |
| Подходит под ограничение 3 месяца | — |

**Вывод:** Strangler Fig Pattern — оптимальный основной подход для эволюции AdScale.

---

## Вариант 4 — постепенная декомпозиция

Постепенная декомпозиция — это более широкий подход, в рамках которого можно использовать и модульный монолит, и Strangler Fig Pattern.

### Домены и потоки для декомпозиции

| Домен / поток |
|---|
| DSP Integration / Bid Gateway |
| Bidding / Auction |
| Event Collection / Statistics |
| Analytics / Reporting |
| Finance / Billing |
| Campaign Management |

### Что выносить в первую очередь

| Критерий приоритизации |
|---|
| То, что находится в critical path |
| То, что создаёт bottleneck |
| То, что требует независимого масштабирования |
| То, что даёт быстрый бизнес-эффект |

**Вывод:** постепенная декомпозиция — это стратегический путь, а Strangler Fig Pattern — практический способ её реализации.

---

# 3. Рекомендуемый подход

Для эволюции **AdScale** выбирается **постепенная декомпозиция монолита с использованием Strangler Fig Pattern**, где монолит сохраняется как стабильная основа, но наиболее нагруженные и рискованные потоки постепенно выносятся в отдельные сервисы.

Внутри оставшейся части монолита вводятся принципы **модульного монолита**:

| Принцип | Описание |
|---|---|
| Явные границы доменов | Модули разделяются по бизнес-областям |
| Контракты между модулями | Взаимодействие происходит через определённые интерфейсы |
| Запрет неявных зависимостей через общую БД | Модули не должны напрямую обращаться к чужим данным |

---

## Почему не переписывать сразу на микросервисы

| Ограничение | Риск |
|---|---|
| Срок 3 месяца | Недостаточно времени для безопасного rewrite |
| Ограниченный бюджет | Не хватит ресурсов на полноценную платформенную трансформацию |
| Отсутствие контейнеризации | Сложнее управлять сервисами и окружениями |
| Отсутствие CI/CD | Высокий риск ручных ошибок |
| Отсутствие SRE | Нет зрелой эксплуатации распределённой системы |
| Отсутствие документации | Высокий риск неверной декомпозиции |
| Big-bang rewrite | Вероятность получить распределённый монолит с худшей latency |

---

## Почему выбран Strangler Fig + постепенная декомпозиция

| Причина | Эффект |
|---|---|
| Безопасная поэтапная миграция | Снижение риска для бизнеса |
| Разгрузка PostgreSQL | Снижение latency и конкуренции за БД |
| Изоляция критичных потоков | Улучшение стабильности RTB-аукциона |
| Подключение новой DSP без остановки бизнеса | Сохранение текущей выручки и клиентов |
| Подготовка к event-driven архитектуре | Основа для целевой архитектуры через год |
| Постепенный вынос сервисов | Управляемая эволюция вместо rewrite |

---

# 4. Соответствие ограничению по срокам

За 3 месяца невозможно безопасно переписать всю платформу. Но можно реализовать ограниченный набор изменений с высоким эффектом.

| Что можно сделать за 3 месяца | Ожидаемый эффект |
|---|---|
| Внедрить кэш для hot data | Снизить latency в critical path |
| Добавить очередь для событий | Разгрузить синхронные операции |
| Разгрузить production PostgreSQL | Снизить contention между RTB, аналитикой и статистикой |
| Подготовить отдельный bidding/runtime слой | Начать изоляцию Auction Engine |
| Ввести маршрутизацию для новой DSP | Управляемо подключить партнёрский трафик |
| Изолировать статистику и аналитику от RTB | Предотвратить влияние отчётов на аукцион |
| Постепенно масштабировать нагруженные компоненты | Увеличить пропускную способность |
| Подготовить платформу к микросервисной архитектуре через год | Создать базу для дальнейшей декомпозиции |

---

# 5. Механизмы безопасного перехода

| Механизм | Назначение |
|---|---|
| Feature flags | Включение и отключение новых путей без деплоя |
| Shadow traffic | Проверка новых компонентов на копии реального трафика |
| Canary rollout | Постепенное включение на малом проценте трафика |
| Dual-write для событий | Переходный период записи в старую и новую модель |
| Постепенное переключение DSP-трафика | Контролируемое подключение новой DSP |
| Rollback на старый путь | Быстрый возврат при проблемах |

---

# 6. Снижение рисков для бизнеса

| Бизнес-риск | Как выбранный подход его снижает |
|---|---|
| Остановка бизнеса | Монолит продолжает обслуживать текущих клиентов |
| Потеря текущих клиентов | Старые потоки остаются рабочими во время миграции |
| Срыв подключения новой DSP | DSP-трафик можно подключать постепенно |
| Рост latency | Hot path разгружается через кэш, очереди и выделение runtime-слоя |
| Отказ всей платформы | Критичные потоки постепенно изолируются |
| Превышение бюджета | Нет необходимости в полном rewrite за 3 месяца |
| Недостаток команды | Подход реалистичен для ограниченной команды |

---

# Итоговое решение

| Пункт | Решение |
|---|---|
| Основной архитектурный подход на 3 месяца | Постепенная декомпозиция |
| Практический паттерн миграции | Strangler Fig Pattern |
| Промежуточная внутренняя организация монолита | Модульный монолит |
| Целевая архитектура на год | Микросервисная event-driven архитектура |
| Подход к миграции | Поэтапный, с feature flags, shadow traffic, canary rollout и rollback |
| Главный приоритет | Разгрузка critical path и безопасное подключение новой DSP |
| Почему это оптимально | Снижает latency, уменьшает риски, не останавливает бизнес и готовит платформу к масштабированию |
