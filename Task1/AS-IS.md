# AS-IS документ текущей архитектуры AdScale

## Общее описание

| Параметр | Описание |
|---|---|
| Система | AdScale |
| Тип системы | Рекламная RTB-платформа |
| Назначение | Позволяет рекламодателям запускать рекламные кампании, управлять бюджетами и ставками, участвовать в аукционах в реальном времени, показывать рекламу пользователям сайтов и приложений, а также собирать статистику и формировать отчётность |
| Текущая реализация | Гибридный монолит |
| Технологии | C++ / Go / Python / Node.js |

## Логические компоненты внутри монолита

| Компонент | Назначение | Особенности / проблемы |
|---|---|---|
| Ad Server | Рекламный сервер подбирает рекламу: принимает HTTP-запросы, определяет пользователей, подбирает кандидатов и передаёт их в Auction Engine | Критичен по latency |
| Auction Engine | Сервер аукциона взвешивает рекламу: применяет бизнес-правила для взвешивания ставок и определения победителя | Наиболее чувствителен к задержкам. Использует синхронные вызовы к базе данных |
| Delivery Service | Сервис выдачи готовит разметку баннера | Формирует HTML/JS-разметку и подготавливает response |
| Statistics Service | Сервис статистики собирает данные по событиям | Записывает клики и показы напрямую в PostgreSQL. Буферизация отсутствует, что создаёт высокую write-нагрузку и конкуренцию с transactional-запросами |
| Financial Service | Финансовый сервис управляет финансами | Выполняет списание средств, выставление счетов и сведение балансов. Работает в той же базе данных |
| Analytics Service | Сервис аналитики собирает аналитику | Выполняет SQL-запросы в production-базу. Read-replica отсутствует. Запросы тяжёлые, что приводит к блокировкам и росту latency |
| Advertiser Dashboard | Рекламный кабинет представляет пользовательский интерфейс для рекламодателей | Управляет рекламой от создания до аналитики. Backend находится внутри монолита. Использует ту же базу данных |

## Результаты анализа

| № | Проблема | Последствия / риски |
|---:|---|---|
| 1 | Все компоненты используют единую PostgreSQL БД | База данных является узким местом и главной точкой отказа |
| 2 | Система работает в одном production-кластере | Отсутствует распределение нагрузки и отказоустойчивость. Отказ кластера приводит к простою всей платформы. Сложно обеспечить SLA 99,9% |
| 3 | Контейнеризация отсутствует | Усложнено масштабирование, выше риск ошибок при деплое, сложнее независимо разворачивать и переносить систему между окружениями |
| 4 | Деплой выполняется вручную | Релизы зависят от ручных операций, высокий риск человеческой ошибки, долгий откат, сложнее делать частые релизы |
| 5 | Кэширование и асинхронные очереди не используются | Все чтения и записи идут напрямую в PostgreSQL |
| 6 | Система нарушает SLA по latency: 110–120 ms при требовании ≤ 100 ms | Нарушение контрактных требований, риск штрафов, проигрыш RTB-аукционов, ухудшение пользовательского и партнёрского опыта |
| 7 | Auction Engine находится в critical path и зависит от синхронных запросов к БД | Любая задержка БД напрямую увеличивает время ответа аукциона. Растёт риск таймаутов, система плохо переносит пики нагрузки |
| 8 | Statistics Service создаёт высокую нагрузку записи без буферизации | Всплески показов и кликов сразу перегружают PostgreSQL, ухудшается работа Auction Engine, возможны задержки записи |
| 9 | Analytics Service выполняет тяжёлые запросы к БД | Отчёты конкурируют с RTB-запросами за ресурсы БД, растёт latency |
| 10 | Критичные и некритичные потоки не разделены | Некритичные операции, например отчёты или запись событий, могут влиять на критичные операции аукциона. Невозможно управлять приоритетами нагрузки |
| 11 | Монолит мешает независимому масштабированию компонентов | Нельзя отдельно масштабировать сервисы, приходится масштабировать весь монолит |
| 12 | Отсутствуют кэш, очереди, read-replicas и HA | Без кэша растёт нагрузка на чтение из БД, без очередей пики событий сразу попадают в БД, без read-replicas нельзя разгрузить чтение, а без HA отказ PostgreSQL приводит к простою |

## Ключевые ограничения текущей архитектуры

| Область | Текущее состояние | Влияние |
|---|---|---|
| База данных | Единая PostgreSQL для всех компонентов | Узкое место, конкуренция нагрузок, единая точка отказа |
| Latency | 110–120 ms | Не выполняется SLA ≤ 100 ms |
| Масштабирование | Масштабируется весь монолит целиком | Невозможно гибко масштабировать отдельные компоненты |
| Отказоустойчивость | Один production-кластер, отсутствует HA | Риск полного простоя платформы |
| Деплой | Ручной | Высокий риск ошибок и долгий rollback |
| Обработка событий | Без очередей и буферизации | Пиковая нагрузка сразу попадает в PostgreSQL |
| Аналитика | Тяжёлые запросы в production-базу | Рост latency и блокировки |
| Кэширование | Отсутствует | Повышенная нагрузка на PostgreSQL |
