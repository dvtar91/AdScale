# ADR-001: Стратегия эволюции архитектуры AdScale — постепенная декомпозиция монолита по Strangler Fig Pattern

## Контекст

AdScale — рекламная RTB-платформа, которая позволяет рекламодателям запускать кампании, управлять ставками и бюджетами, участвовать в аукционах в реальном времени, показывать рекламу пользователям сайтов и приложений, собирать статистику и формировать отчётность.

Текущая реализация системы — **гибридный монолит** на **C++ / Go / Python / Node.js**.

Внутри монолита находятся следующие логические компоненты:

- **Ad Server** — принимает HTTP-запросы, определяет пользователя, подбирает кандидатов.
- **Auction Engine** — применяет бизнес-правила, ставки и выбирает победителя RTB-аукциона.
- **Delivery Service** — формирует HTML/JS-разметку и response.
- **Statistics Service** — записывает показы и клики.
- **Financial Service** — управляет списаниями, балансами и счетами.
- **Analytics Service** — выполняет аналитические SQL-запросы.
- **Advertiser Dashboard** — пользовательский интерфейс и backend рекламного кабинета.

Все компоненты используют одну production-базу **PostgreSQL**.

Система должна быть подготовлена к интеграции с новой крупной DSP. Новая интеграция увеличит трафик примерно на **20%**:

| Метрика | Текущее значение / цель |
|---|---:|
| Текущая нагрузка | 15 000 RPS |
| Ожидаемая нагрузка после подключения DSP | 18 000 RPS |
| Текущий P95 latency | 110–120 ms |
| Требуемый P95 latency | ≤ 100 ms |
| Целевой P95 latency для новой DSP | ≤ 80 ms |
| Целевой горизонт через год | до 50 000 RPS |
| Целевой SLA доступности через год | 99,9% |

Сейчас система уже не выполняет SLA по latency на текущей нагрузке. Подключение новой DSP без архитектурных изменений создаёт высокий риск деградации: рост latency, таймауты, проигрыш RTB-аукционов, штрафы и потеря выручки.

---

## Архитектурные драйверы

Ключевые драйверы решения:

| Категория | Драйвер |
|---|---|
| Performance / Latency | RTB-response P95 должен быть ≤ 100 ms, целевой уровень для новой DSP — ≤ 80 ms |
| Scalability | Система должна выдержать минимум 18 000 RPS через 3 месяца |
| Reliability | Bidding service не должен давать массовых таймаутов и сбоев |
| Fault Isolation | Статистика, аналитика и отчётность не должны ухудшать RTB hot path |
| Availability | Система должна продолжать обрабатывать bid requests при отказе некритичных компонентов |
| Data Consistency | Финансовые операции, балансы и списания должны оставаться корректными |
| Evolutionary Modifiability | Полный rewrite невозможен, нужна постепенная миграция |
| Deployability | Ручной деплой должен быть заменён или минимизирован безопасным rollout/rollback |
| Observability | Нужны метрики, логи и трассировка critical path |

---

## Ограничения

### Технические ограничения

- Текущая система — гибридный монолит.
- Все компоненты используют одну PostgreSQL-инстанцию.
- Нет read-replicas.
- Нет PostgreSQL HA.
- Нет кэширования.
- Нет очередей и буферизации событий.
- Statistics Service пишет события напрямую в PostgreSQL.
- Analytics Service выполняет тяжёлые запросы в production-базу.
- Нет контейнеризации.
- Деплой выполняется вручную.
- Один production-кластер.
- Нет выделенного SRE.
- Архитектура недостаточно документирована.

### Бизнес-ограничения

- Промежуточное решение должно быть реализовано за **3 месяца**.
- Полная переработка системы невозможна.
- Нельзя останавливать бизнес.
- Нужно сохранить текущих клиентов и DSP-интеграции.
- Новая DSP стратегически важна.
- Бюджет ограничен.
- Команда ограничена примерно **35 сотрудниками**.
- Через год ожидается рост до **50 000 RPS**.

---

## Проблема

Текущий монолит смешивает критичные и некритичные потоки.

В одном контуре и на одной PostgreSQL конкурируют:

- RTB-аукцион;
- чтение ставок и кампаний;
- запись показов и кликов;
- финансовые транзакции;
- тяжёлые аналитические отчёты;
- CRUD-операции рекламного кабинета.

Наиболее критичные потоки:
DSP Partner:
 - Ad Server
 - Auction Engine
 - Delivery Service
 - Bid Response


Сейчас **Auction Engine** находится в hot path и зависит от синхронных запросов к PostgreSQL. Любая задержка БД напрямую увеличивает время ответа аукциона.

Также **Statistics Service** создаёт высокую write-нагрузку, потому что пишет события напрямую в PostgreSQL без очереди. **Analytics Service** выполняет тяжёлые SQL-запросы в production-базу, что создаёт блокировки и конкуренцию за ресурсы.

Итоговая проблема: система не имеет запаса производительности и отказоустойчивости для подключения новой DSP.

---

## Рассмотренные альтернативы

### Альтернатива 1. Оставить текущий монолит без существенных изменений

**Описание:**  
Система остаётся в текущем виде. Новая DSP подключается к существующему Ad Server / Auction Engine. Масштабирование выполняется за счёт увеличения ресурсов текущего production-кластера и PostgreSQL.

**Плюсы:**

- Минимальные изменения в коде.
- Самый быстрый краткосрочный time-to-market.
- Не требуется менять модель разработки и эксплуатации.
- Меньше рисков функциональных регрессий на уровне бизнес-логики.

**Минусы:**

- SLA уже нарушается: P95 latency 110–120 ms при требовании ≤ 100 ms.
- Рост до 18 000 RPS усилит нагрузку на Auction Engine и PostgreSQL.
- Единая PostgreSQL остаётся bottleneck и single point of failure.
- Статистика, аналитика и dashboard продолжат влиять на RTB-аукцион.
- Нет независимого масштабирования компонентов.
- Нет подготовки к 50 000 RPS через год.
- Высокий риск штрафов, таймаутов и проигрыша RTB-аукционов.

**Вывод:**  
Вариант отклонён. Он не закрывает P0-драйверы: latency, scalability, reliability и fault isolation.

---

### Альтернатива 2. Немедленный rewrite в микросервисы

**Описание:**  
За короткий срок переписать платформу в микросервисную архитектуру: выделить отдельные сервисы для DSP Gateway, Ad Serving, Auction, Statistics, Finance, Analytics, Campaign Management, внедрить database-per-service, Kafka, Redis, Kubernetes, CI/CD и observability.

**Плюсы:**

- Потенциально лучшая горизонтальная масштабируемость.
- Возможность database-per-service.
- Хорошая fault isolation при зрелой реализации.
- Соответствие целевой архитектуре через год.
- Упрощение независимого масштабирования Ad Server, Auction Engine, Statistics и Analytics.

**Минусы:**

- Срок 3 месяца недостаточен для безопасного полного rewrite.
- Высокий риск получить distributed monolith.
- Возможен рост latency из-за сетевых вызовов между сервисами.
- Нужны зрелые CI/CD, контейнеризация, мониторинг, SRE-практики.
- Сейчас нет выделенного SRE и нет контейнеризации.
- Появляется сложность distributed tracing, retries, timeouts, circuit breakers.
- Финансовая консистентность потребует аккуратного проектирования транзакционных границ, идемпотентности и, возможно, Saga-подходов.
- Высокий риск сорвать интеграцию новой DSP и повлиять на текущих клиентов.

**Вывод:**  
Вариант отклонён как немедленная стратегия. Микросервисы признаются целевым направлением на горизонт года, но не подходят как big-bang rewrite в текущих условиях.

---

### Альтернатива 3. Модульный монолит

**Описание:**  
Система остаётся единым deployment unit, но внутри монолита вводятся чёткие модульные границы: Campaign, Bidding, Delivery, Statistics, Finance, Analytics, Dashboard. Взаимодействие между модулями происходит через публичные интерфейсы. Прямые зависимости через общую БД постепенно ограничиваются.

**Плюсы:**

- Реалистично для текущей команды и сроков.
- Меньше операционной сложности, чем у микросервисов.
- Улучшает maintainability.
- Помогает подготовиться к будущей декомпозиции.
- Позволяет формализовать границы доменов.
- Снижает риск хаотичных изменений в монолите.

**Минусы:**

- Не даёт полноценного независимого масштабирования.
- Не решает полностью проблему общей PostgreSQL.
- Не обеспечивает достаточную fault isolation для RTB hot path.
- Не устраняет влияние тяжёлой аналитики и статистики на базу без дополнительных инфраструктурных изменений.
- Не позволяет отдельно масштабировать Auction Engine, если он остаётся в общем deployment unit.

**Вывод:**  
Вариант полезен как часть стратегии, но недостаточен сам по себе. Его нужно применять внутри оставшейся части монолита вместе с постепенным выносом критичных потоков.

---

### Альтернатива 4. Постепенная декомпозиция по Strangler Fig Pattern

**Описание:**  
Монолит продолжает работать, но наиболее критичные и нагруженные потоки постепенно выносятся в отдельные сервисы. Трафик маршрутизируется через gateway/роутинг-слой. Новые сервисы подключаются через feature flags, shadow traffic, canary rollout и rollback.

Приоритет выноса:

1. **DSP Bid Gateway / DSP Integration** — изоляция нового партнёрского трафика.
2. **RTB hot path / Bidding runtime** — разгрузка Auction Engine, снижение зависимости от PostgreSQL.
3. **Event Collection / Statistics** — асинхронная запись показов, кликов и auction events через очередь.
4. **Analytics / Reporting** — вынос тяжёлых запросов из production PostgreSQL.
5. **Finance / Billing** — аккуратная изоляция финансового домена с сохранением консистентности.
6. **Campaign Management** — дальнейшая декомпозиция dashboard и управления кампаниями.

**Плюсы:**

- Снижает риск big-bang rewrite.
- Позволяет не останавливать бизнес.
- Даёт быстрый эффект на critical path.
- Позволяет подключать новую DSP постепенно.
- Позволяет применять shadow traffic и canary.
- Поддерживает rollback на старый путь.
- Позволяет внедрять Redis, Kafka, read-replicas и HA поэтапно.
- Создаёт основу для целевой микросервисной event-driven архитектуры.
- Соответствует ограничению в 3 месяца.

**Минусы:**

- На переходном этапе появляется гибридная архитектура.
- Требуются адаптеры между монолитом и новыми сервисами.
- Возможна временная дубликация логики.
- Нужна дисциплина управления контрактами и границами.
- Появляется дополнительная операционная сложность.
- Требуется observability, чтобы контролировать новый распределённый контур.

**Вывод:**  
Вариант выбран как основной. Он лучше всего балансирует срочные требования по latency/scalability и ограничения по срокам, бюджету и организационной зрелости.

- нужно быстро снизить риск деградации RTB;
- нельзя останавливать бизнес;
- нельзя переписать платформу за 3 месяца;
- нужно начать движение к микросервисной event-driven архитектуре;
- нужно подготовиться к 50 000 RPS через год.

## Решение

Принято решение развивать AdScale по стратегии:

> **Постепенная декомпозиция текущего гибридного монолита с использованием Strangler Fig Pattern.**

В горизонте **3 месяцев** фокус делается не на полном переходе к микросервисам, а на стабилизации и разгрузке **RTB critical path**.

В горизонте **1 года** целевым состоянием является **микросервисная event-driven архитектура** с независимым масштабированием сервисов, разделением баз данных и инфраструктурой для SLA 99,9%.

---

## Принципы реализации решения

### 1. Монолит не переписывается полностью

Существующий монолит продолжает обслуживать текущих клиентов и существующие DSP-интеграции. Новые компоненты внедряются рядом с ним.

---

### 2. Внутри монолита вводятся принципы модульного монолита

Оставшиеся части монолита должны быть структурированы по доменным границам:

- Bidding;
- Campaign Management;
- Finance;
- Statistics;
- Analytics;
- Delivery;
- Dashboard.

Правила:

- каждый модуль имеет явные публичные интерфейсы;
- запрещаются новые неявные зависимости через общую БД;
- новые интеграции оформляются через API, события или фасады;
- владение данными постепенно закрепляется за доменными областями.

---

### 3. Первым выделяется контур новой DSP и RTB hot path

Приоритетные компоненты для выделения:

| Приоритет | Компонент / поток | Причина |
|---:|---|---|
| 1 | DSP Bid Gateway | Изоляция новой DSP, rate limiting, canary, routing, timeout budget |
| 2 | Кэш hot data для Auction / Ad Server | Снижение синхронных запросов в PostgreSQL |
| 3 | Event Collector + очередь | Удаление записи статистики из synchronous path |
| 4 | Analytics read model / DWH или read-replica | Удаление тяжёлых отчётов из production PostgreSQL |
| 5 | Bidding runtime / Auction Service | Независимое масштабирование critical path |
| 6 | Finance / Billing service boundary | Сохранение финансовой консистентности и отделение от RTB |

---

### 4. Используются инфраструктурные элементы целевой архитектуры

В рамках эволюции должны быть внедрены:

- **API Gateway / DSP Bid Gateway** для маршрутизации и контроля трафика;
- **Redis Cluster** для hot data: кампании, ставки, таргетинг, budget snapshots;
- **Kafka** или совместимый брокер для событий: impressions, clicks, auction results, billing events;
- **read-replicas / analytics store** для разгрузки production PostgreSQL;
- **PostgreSQL HA** для критичных OLTP-данных;
- **Observability stack**: метрики, логи, трассировка, SLA/SLO dashboards;
- **Feature flags** для безопасного включения новых путей;
- **Canary rollout** и rollback;
- контейнеризация новых сервисов.

---

### 5. Миграция должна быть безопасной

Для каждого нового сервиса или нового потока применяются:

- feature flags;
- shadow traffic;
- canary rollout;
- поэтапное переключение DSP-трафика;
- fallback на старый путь;
- мониторинг latency, error rate, timeout rate и business metrics;
- нагрузочное тестирование перед увеличением доли трафика.

---

## Целевая архитектурная траектория

### Этап 1. 0–3 месяца: стабилизация critical path

Цель: безопасно подключить новую DSP и выдержать **18 000 RPS**.

Основные изменения:

- внедрить DSP routing / Bid Gateway для новой DSP;
- добавить Redis-кэш для hot data;
- убрать часть синхронных чтений PostgreSQL из Auction Engine;
- внедрить очередь для показов, кликов и auction events;
- перенести тяжёлую аналитику на read-replica или отдельное аналитическое хранилище;
- ввести базовую observability для RTB path;
- подготовить canary и rollback.

---

### Этап 2. 3–6 месяцев: выделение первых сервисов

Цель: начать физическую изоляцию наиболее нагруженных областей.

Кандидаты:

- DSP Bid Gateway;
- Event Collector;
- Statistics Processor;
- Reporting Service;
- Cache update workers;
- частично Bidding runtime.

---

### Этап 3. 6–12 месяцев: целевая микросервисная event-driven архитектура

Цель: подготовиться к **50 000 RPS** и SLA **99,9%**.

Кандидаты:

- Auction Service;
- Ad Serving Service;
- Campaign Service;
- Bid Strategy Service;
- Creative Service;
- User Profile Service;
- Budget Guard Service;
- Finance Service;
- Billing Service;
- Payment Integration Service;
- Analytics Processor;
- Reporting Service.

Принцип хранения данных:

> Каждый сервис владеет своими данными. Общая production PostgreSQL больше не используется как универсальное хранилище для всех доменов.

---

## Последствия

### Положительные последствия

- Снижается риск деградации при подключении новой DSP.
- RTB hot path постепенно изолируется от статистики, аналитики и dashboard.
- PostgreSQL разгружается за счёт кэша, очередей, read-replicas и выделения хранилищ.
- Появляется возможность независимо масштабировать наиболее нагруженные компоненты.
- Миграция не требует остановки бизнеса.
- Можно постепенно переключать трафик и откатываться при проблемах.
- Создаётся база для event-driven архитектуры.
- Архитектура становится готовой к росту до 50 000 RPS.
- Улучшается observability и управляемость SLA.
- Уменьшается радиус поражения при сбоях некритичных компонентов.

---

### Отрицательные последствия

- На переходном этапе система становится сложнее: одновременно существуют монолит и новые сервисы.
- Возникает необходимость поддерживать адаптеры между монолитом и сервисами.
- Возможна временная дубликация логики и данных.
- Появляется дополнительная сложность эксплуатации Kafka, Redis, gateway и новых сервисов.
- Требуется дисциплина управления контрактами API и событий.
- Команде нужно развивать DevOps/SRE-практики.
- Для финансового домена потребуется аккуратное проектирование идемпотентности, транзакционных границ и сверок.

---

### Риски и меры снижения

| Риск | Вероятность / влияние | Митигирующие меры |
|---|---|---|
| Новый RTB path ухудшит latency из-за сетевых вызовов | Средняя / высокая | Shadow traffic, нагрузочные тесты, timeout budget, p95/p99 monitoring, co-location сервисов |
| Kafka или Redis станут новой точкой отказа | Средняя / высокая | Кластеризация, replication, monitoring, fallback, capacity planning |
| Команда не справится с операционной сложностью | Средняя / высокая | Поэтапное внедрение, runbooks, обучение, минимальный platform team / SRE ownership |
| Дублирование логики приведёт к расхождениям | Средняя / средняя | Контракты, contract tests, ownership, постепенное удаление старой логики |
| Dual-write вызовет несогласованность данных | Средняя / высокая | Outbox pattern, idempotency keys, reconciliation jobs, audit logs |
| Финансовые списания станут неконсистентными | Низкая / критичная | Finance Service как владелец ledger, ACID PostgreSQL, идемпотентные операции, сверки |
| Аналитика продолжит нагружать production-БД | Средняя / высокая | Запрет heavy queries на master, read-replica, ClickHouse/DWH, query governance |
| Canary новой DSP повлияет на текущих клиентов | Низкая / высокая | Изоляция DSP-трафика, per-partner rate limits, kill switch |
| Невозможно достичь 80 ms для новой DSP | Средняя / высокая | Кэш hot data, исключение sync DB из hot path, profiling, оптимизация Auction Engine |

## Критерии успешности решения

Решение считается успешным для горизонта 3 месяцев, если:

- система выдерживает **18 000 RPS** без массовых таймаутов;
- P95 RTB-response ≤ **100 ms**;
- для новой DSP достигнут целевой уровень, близкий к **≤ 80 ms**, либо есть подтверждённый план оптимизации;
- Statistics Service больше не пишет весь поток событий напрямую в production PostgreSQL синхронно;
- тяжёлые отчёты не выполняются на master PostgreSQL;
- есть dashboard по latency, RPS, error rate, timeout rate, Kafka lag, Redis latency/cache hit ratio;
- есть возможность canary rollout и rollback для новой DSP;
- отказ аналитики или задержка обработки статистики не приводит к остановке bid responses.

Решение считается успешным для горизонта года, если:

- архитектура поддерживает до **50 000 RPS**;
- доступность приближается к SLA **99,9%**;
- RTB, события, финансы и аналитика разделены по сервисам и хранилищам;
- основные сервисы контейнеризированы и разворачиваются через CI/CD;
- каждый критичный сервис имеет observability, health checks, runbooks;
- новая DSP-интеграция может добавляться без существенных изменений core bidding logic.

## Пересмотр ADR

ADR должен быть пересмотрен через **6 месяцев** или раньше, если произойдёт одно из событий:

- нагрузка превысит **18 000 RPS** раньше ожидаемого срока;
- новая DSP потребует latency ниже **80 ms**;
- число DSP-партнёров значительно увеличится;
- появится выделенная SRE/platform-команда;
- будет внедрён Kubernetes и зрелый CI/CD;
- появится необходимость вынести финансовый домен в отдельный сервис раньше плана;
- текущая стратегия не позволит выполнить SLA по latency или availability.

## Итог

AdScale будет развиваться по стратегии:

> **Strangler Fig Pattern + постепенная декомпозиция + модульный монолит внутри оставшейся части системы.**

Это решение обеспечивает баланс между срочными бизнес-требованиями подключения новой DSP, снижением latency, fault isolation, ограничениями команды и целевой архитектурой на год.
