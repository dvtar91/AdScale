# Архитектурные драйверы AdScale после интеграции с новой DSP

## 1. Функциональные требования к системе после интеграции

| ID | Функциональное требование | Описание |
|---|---|---|
| FR-01 | Принимать bid requests от новой DSP | Система должна принимать запросы на ставки от новой крупной DSP-площадки |
| FR-02 | Обрабатывать bid requests в реальном времени | Каждый bid request должен проходить через Ad Server и Auction Engine без блокировки критичного потока |
| FR-03 | Выполнять подбор рекламы | Ad Server должен определять пользователя, применять таргетинг и подбирать подходящие кампании |
| FR-04 | Рассчитывать победителя RTB-аукциона | Auction Engine должен применять бизнес-правила, ставки, ограничения бюджета и выбирать победителя |
| FR-05 | Формировать bid response | Система должна возвращать корректный bid response DSP-партнёру в рамках SLA |
| FR-06 | Формировать разметку | Delivery Service должен формировать HTML/JS-разметку или другой response |
| FR-07 | Регистрировать показы рекламы | Система должна фиксировать события показов для статистики, аналитики и биллинга |
| FR-08 | Регистрировать клики | Система должна фиксировать события кликов по рекламе |
| FR-09 | Управлять рекламными кампаниями | Рекламодатель должен иметь возможность создавать, изменять, включать и отключать кампании |
| FR-10 | Управлять ставками | Рекламодатель должен иметь возможность изменять ставки и стратегии участия в аукционе |
| FR-11 | Управлять бюджетами | Система должна поддерживать пополнение, списание и контроль бюджетов рекламодателей |
| FR-12 | Выполнять финансовые операции | Financial Service должен обеспечивать списание средств, выставление счетов и управление балансами |
| FR-13 | Интегрироваться с платёжным шлюзом | Система должна обрабатывать пополнение рекламных бюджетов через внешний payment gateway |
| FR-14 | Предоставлять аналитику и отчётность | Рекламодатель должен видеть статистику по кампаниям, показам, кликам, расходам и эффективности |
| FR-15 | Поддерживать несколько DSP-партнёров | Система должна сохранять текущие DSP-интеграции и добавить новую DSP без деградации текущих клиентов |
| FR-16 | Обрабатывать поток событий | Система должна принимать и обрабатывать постоянный поток событий показов, кликов и результатов аукционов |
| FR-17 | Разделять критичные и некритичные потоки | RTB-аукцион должен быть отделён от статистики, аналитики и отчётности |
| FR-18 | Поддерживать graceful degradation | При деградации некритичных компонентов система должна продолжать обрабатывать bid requests |
| FR-19 | Поддерживать мониторинг SLA | Система должна собирать метрики latency, RPS, ошибок, таймаутов и доступности |
| FR-20 | Поддерживать постепенную миграцию | Новые компоненты должны внедряться без полной остановки и переписывания монолита |

## 2. Атрибуты качества с приоритизацией

**Приоритеты:**

- **P0 — критично для интеграции DSP и непрерывности бизнеса**
- **P1 — важно для стабильной работы в горизонте 3 месяцев**
- **P2 — важно для целевой архитектуры на горизонте 1 года**

### 2.1. P0 — критичные атрибуты качества

| Атрибут качества | Требование | Почему P0 |
|---|---|---|
| **Performance / Latency** | P95 RTB-response должен быть ≤ 100 ms, целевой уровень для новой DSP — ≤ 80 ms | Новая DSP предъявляет жёсткие SLA, нарушение latency ведёт к штрафам и проигрышу аукционов |
| **Scalability** | Система должна выдерживать минимум 18 000 RPS через 3 месяца | Интеграция увеличит трафик на 20% к текущим 15 000 RPS |
| **Reliability** | Bidding service не должен давать массовых сбоев, таймаутов и ошибок при пиковой нагрузке | Сбой аукциона напрямую приводит к потере показов и выручки |
| **Availability** | Система должна сохранять доступность при отказе отдельных некритичных компонентов | DSP предъявляет требования к downtime, простои приводят к штрафам |
| **Fault Isolation** | Некритичные потоки — аналитика, статистика, отчётность — не должны ухудшать RTB hot path | Сейчас эти потоки конкурируют за PostgreSQL и повышают latency |
| **Data Consistency для финансов** | Балансы и списания должны оставаться корректными | Ошибки в финансах приводят к прямым денежным потерям и конфликтам с рекламодателями |

### 2.2. P1 — важные атрибуты качества для горизонта 3 месяцев

| Атрибут качества | Требование | Почему P1 |
|---|---|---|
| **Elasticity** | Наиболее нагруженные части системы должны масштабироваться независимо или частично независимо | Нужно масштабировать Ad Server, Auction Engine и обработку событий без масштабирования всего монолита |
| **Throughput** | Система должна обрабатывать увеличенный поток bid requests, показов и кликов | Рост RPS увеличит не только аукционы, но и event stream |
| **Observability** | Должны быть метрики, логи и трассировка по critical path | Без наблюдаемости невозможно управлять SLA и находить bottleneck |
| **Recoverability** | Система должна быстро восстанавливаться после сбоя или неуспешного релиза | Сейчас ручной деплой и одна БД увеличивают время восстановления |
| **Deployability** | Деплой должен стать более безопасным, воспроизводимым и откатываемым | Для постепенной миграции нужны частые и безопасные изменения |
| **Maintainability** | Архитектура должна быть задокументирована, а зоны ответственности компонентов — понятны | Сейчас архитектура не задокументирована, что повышает риск изменений |
| **Backward Compatibility** | Интеграция новой DSP не должна ломать существующих партнёров и клиентов | Нельзя потерять текущих клиентов при подключении нового партнёра |

### 2.3. P2 — атрибуты качества для целевого состояния через год

| Атрибут качества | Требование | Почему P2 |
|---|---|---|
| **Horizontal Scalability** | Система должна поддерживать до 50 000 RPS | Через год ожидается кратный рост нагрузки |
| **High Availability** | Целевой SLA — 99,9% | Для зрелой DSP-платформы требуется архитектурная отказоустойчивость |
| **Modifiability** | Новые DSP-интеграции должны добавляться без существенных изменений core bidding logic | Бизнес планирует мультиплатформенные интеграции |
| **Event-driven Processing** | События показов, кликов, биллинга и аналитики должны обрабатываться асинхронно | Это нужно для масштабируемости и разгрузки hot path |
| **Multi-region Readiness** | Архитектура должна быть готова к мультирегиональному развёртыванию | Целевая архитектура предполагает мультирегиональность |
| **Operational Maturity** | Должны быть CI/CD, автоматизированный rollback, мониторинг SLA, runbooks | Для SLA 99,9% нужна зрелая эксплуатационная модель |
| **Data Scalability** | OLTP, event stream и аналитика должны быть разделены | Единая PostgreSQL не выдержит 50 000 RPS и большие объёмы событий |

## 4. Технические ограничения

| ID | Ограничение | Влияние на архитектуру |
|---|---|---|
| TC-01 | Текущая система — гибридный монолит C++ / Go / Python / Node.js | Нельзя быстро перейти к чистой микросервисной архитектуре, нужна постепенная декомпозиция |
| TC-02 | Основная БД — PostgreSQL, одна инстанция | Требуется разгрузка БД, read-replicas, HA или выделение отдельных хранилищ |
| TC-03 | Кэширование отсутствует | Необходимо внедрять кэш для hot data: кампании, ставки, таргетинг |
| TC-04 | Очереди отсутствуют | Нужно добавлять асинхронный слой для событий показов, кликов и аналитики |
| TC-05 | Контейнеризация отсутствует | Масштабирование и деплой компонентов ограничены |
| TC-06 | Деплой выполняется вручную | Любые изменения несут высокий операционный риск |
| TC-07 | Один production-кластер | Нет инфраструктурной отказоустойчивости и георезервирования |
| TC-08 | Нет read-replicas | Вся read-нагрузка идёт в master PostgreSQL |
| TC-09 | Нет HA для PostgreSQL | Отказ БД приводит к отказу всей платформы |
| TC-10 | Analytics Service работает по production-БД | Нельзя масштабировать аналитику без влияния на RTB |
| TC-11 | Statistics Service пишет события напрямую в БД | Event stream создаёт write pressure на PostgreSQL |
| TC-12 | Архитектура не задокументирована | Повышается риск ошибок при миграции |
| TC-13 | Нет выделенного SRE | Ограничена зрелость эксплуатации, мониторинга, incident response и SLA-процессов |
| TC-14 | Гибридный стек | Усложняет унификацию observability, CI/CD, деплоя и стандартов разработки |

## 7. Главный архитектурный вывод

Ключевыми драйверами для AdScale являются:

1. **Performance** — нужно снизить P95 с текущих 110–120 ms до ≤ 100 ms, а для новой DSP стремиться к ≤ 80 ms.
2. **Scalability** — необходимо выдержать рост до 18 000 RPS за 3 месяца и подготовиться к 50 000 RPS через год.
3. **Fault Isolation** — критичный RTB-поток должен быть отделён от статистики, аналитики и dashboard-нагрузки.
4. **Reliability / Availability** — bidding service и PostgreSQL не должны быть единственными точками отказа.
5. **Evolutionary Modifiability** — систему нельзя переписать полностью, поэтому нужна постепенная миграция.
6. **Operational Safety** — ручной деплой и отсутствие контейнеризации должны быть устранены или снижены как риск.

Главный фокус промежуточной архитектуры на 3 месяца должен быть не в полном переходе на микросервисы, а в **стабилизации и разгрузке RTB critical path**.
